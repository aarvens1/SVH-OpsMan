### Spec

#### Design overview

Three skills replacing the current single-file approach:

| Skill | Slash | Role |
|-------|-------|------|
| `gemini-handoff` (modified) | `/gemini-handoff` | Creates Obsidian note, status: draft. No longer writes `.gemini/handoff.md`. |
| `handoff-queue` (new) | `/handoff-queue` | Scans `Handoffs/` for `status: ready`, writes spec to `.gemini/handoff.md`, flips to in-progress. |
| `handoff-receive` (new) | `/handoff-receive` | Picks up `to-claude.md`, integrates, updates note to done. |

`.gemini/handoff.md` becomes ephemeral ‚Äî the active work slot only, written by `handoff-queue`, consumed by Gemini. The Obsidian notes are the source of truth and the queue.

#### Obsidian note spec

**Vault path:** `Handoffs/YYYY-MM-DD-HH:MM-<task-slug>.md`

**Frontmatter:**
```yaml
---
date: YYYY-MM-DD
created: YYYY-MM-DDTHH:MM:SS
skill: gemini-handoff
status: draft
target: dev | docs | research
gemini_skill: claude-handoff
task_slug: <kebab-case-slug>
tags: [handoff, gemini]
related_briefing: "[[Briefings/Daily/YYYY-MM-DD]]"
---
```

**Status lifecycle:** `draft ‚Üí ready ‚Üí in-progress ‚Üí done`

**Note body structure:**
```markdown
# Handoff ‚Äî <task-slug>

*Queued from [[Briefings/Daily/YYYY-MM-DD]] at HH:MM*

## Summary
[One paragraph: what was asked for, what Gemini will do, which account and skill]

## How to push this handoff
1. Review the spec below ‚Äî edit anything that needs adjusting
2. Change `status: draft` to `status: ready` in the frontmatter above
3. Run `/handoff-queue` in Claude Code

## Spec

[Full sanitized spec ‚Äî same content that would go to .gemini/handoff.md. Identical format.]

---

## Result

*Pending*
```

When `handoff-receive` runs, it replaces `*Pending*` with a summary of what Gemini did and any files changed.

#### New skill: handoff-queue/SKILL.md

**Trigger phrases:** "run ready handoffs", "push ready handoffs", "process handoffs", `/handoff-queue`

**When to use:** When Aaron has marked one or more handoffs `status: ready` in Obsidian and wants to push them to Gemini.

**Steps:**
1. List all files in `Handoffs/` from the vault. Read frontmatter for each.
2. Collect all notes where `status: ready`, sorted by `created` ascending (oldest first).
3. If none found: reply "No handoffs with status: ready. Update a handoff's frontmatter to `status: ready` in Obsidian, then re-run."
4. For each ready handoff (process one at a time ‚Äî do not batch):
   a. Read the full note to extract the `## Spec` section content.
   b. Write that content to `.gemini/handoff.md` in the project root (overwrite).
   c. Update the Obsidian note frontmatter: `status: draft` ‚Üí `status: in-progress`.
   d. Report: "Pushed: `[[Handoffs/YYYY-MM-DD-HH:MM-slug]]` ‚Äî fire **Gemini [target] account**, use `[gemini_skill]` skill. When done, run `/handoff-receive`."

### Handoff Workflow

Each Gemini handoff is tracked as an Obsidian note in `Handoffs/` with a status lifecycle:

`draft` ‚Üí `ready` ‚Üí `in-progress` ‚Üí `done`

**Queueing a handoff:** Run `/gemini-handoff` ‚Äî Claude creates a note in `Handoffs/`, adds a link to your daily Activity Log, and prompts you to review the spec. Multiple handoffs can queue as `draft` without blocking each other.

**Pushing to Gemini:** Change the note's `status` to `ready` in Obsidian, then run `/handoff-queue`. Claude writes the spec to `.gemini/handoff.md` and tells you which account and skill to use. If multiple are ready, they process one at a time ‚Äî run `/handoff-queue` again after each receive.
